# Muphys Graupel GPU Optimization Report (JAX → XLA / IREE)

> Optimizing JAX-compiled graupel microphysics via XLA and IREE compiler backends towards GT4Py DaCe GPU performance on NVIDIA GH200 and AMD MI300A.
> Target: reduce per-iteration time from ~51ms to <10ms (GT4Py DaCe GPU backend baseline on GH200).
> JAX alone cannot reach the target; the strategy relies on custom XLA and IREE compiler passes to re-roll unrolled loops into efficient GPU kernels.

---

## The Problem: Why JAX Can't Reach the Target Natively

The graupel microphysics precipitation scan loops over **90 vertical levels**, where each level's output feeds into the next (a sequential dependency or "carry"). In Python/JAX this looks like:

```python
for k in range(90):
    q_k = q[k]                     # read level k
    out_k, carry = compute(q_k, carry)  # compute with carry from k-1
    result[k] = out_k              # store result
```

When JAX traces this loop, it **unrolls all 90 iterations** into a flat graph. XLA then compiles each iteration as a **separate GPU kernel** (~186 kernels total). Each kernel does very little work (one level for 327,680 cells), but the GPU kernel launch overhead (~5-10us per launch) dominates. The result: **~51ms** runtime, of which most is launch overhead, not computation.

The GT4Py DaCe GPU backend achieves **<10ms** on GH200 for the same physics because DaCe generates a single GPU kernel that loops internally over all 90 levels. Our optimization work aims to achieve the same: **re-roll the unrolled loop into 1-2 GPU kernels** via custom compiler passes operating on XLA's HLO or IREE's MLIR.

## Key Technical Contribution: Custom Primitive + StableHLO Injection

A reusable technique for replacing arbitrary parts of JAX-traced computation with hand-optimized or script-generated StableHLO, **transparently within JAX's normal JIT pipeline**.

**How it works:**

1. **Define a JAX custom primitive** via `jax.extend.core.Primitive("name")` with abstract eval (shape/dtype inference) and eager-mode fallback implementation
2. **Register a custom MLIR lowering** via `mlir.register_lowering(primitive, lowering_fn)` — the lowering function loads a pre-generated StableHLO module from disk and merges it into the JAX compilation module using `mlir.merge_mlir_modules()`
3. The merged module becomes part of XLA's normal compilation — XLA sees it as native HLO and applies all standard optimizations (fusion, scheduling, memory planning)

**Why this matters:**

- No custom C++/CUDA code required — works entirely in Python + StableHLO text
- Composable with all JAX transforms (grad, vmap, pmap)
- The injected module can be generated by any tool (hand-written, script-generated, exported from another framework)
- Transparent to users of the function — it's just a regular JAX function call

**Implementation files:**

- [core/optimized_graupel.py](../core/optimized_graupel.py) — combined graupel primitive (q_t_update + precip)
- [core/optimized_precip.py](../core/optimized_precip.py) — precipitation-only primitive
- [tools/generate_unrolled_transposed.py](../tools/generate_unrolled_transposed.py) — script to generate the StableHLO modules
- [stablehlo/graupel_combined.stablehlo](../../stablehlo/graupel_combined.stablehlo) — pre-generated combined module (~1.8MB)

## Optimization Tracks

### Track 1: JAX-Level Optimizations (Dec 2025 – Jan 2026)

| Approach | Result |
|:---|:---|
| Buffer donation | No improvement |
| Tiling / unrolling / Pallas | No improvement — JAX's scan primitives don't help with kernel count |
| Triton-JAX integration | Triton kernel faster, but DLPack interchange overhead negates gains |
| Memory layout transposition (ncells,nlev) → (nlev,ncells) | Significant improvement — prerequisite for StableHLO injection |
| Fused q_t_update (lax.select/lax.pow) | ~80 kernels → 2 fused kernels (confirmed via nsys) |

### Track 2: StableHLO Injection (Feb 2026)

- Custom primitive mechanism (see above) to inject pre-generated StableHLO
- First applied to precipitation scans only: 51ms → 35ms
- Extended to combined module (q_t_update + precip): 35ms → 29ms on GH200, **23ms on MI300A**
- Benefits only visible after transpose fix (Track 1)

### Track 3: XLA Compiler Pass — LoopifyUnrolledSlices (Mar 2026)

Custom XLA HLO pass that detects the unrolled 90-level `slice → compute → concat` pattern and re-rolls it into a loop or single-kernel serial scan.

- Built JAX and XLA from source to integrate the pass
- Two modes: `kWhileLoop` (~6 kernels, 35ms) and `kSerialScan` (1 kernel, 33ms)
- All 11 output concatenates grouped into one loop with 11 accumulators + 19 carry pairs
- Correctness verified against reference data
- Still ~4ms slower than StableHLO injection baseline
- See [INTEGRATION.md](../xla_passes/INTEGRATION.md) for build/integration details

### Track 4: IREE (Mar 2026)

**IREE CUDA Backend:**
- Patched IREE CUDA bug; functional but slower than XLA due to JIT memory errors requiring function splitting

**IREE HIP/ROCm Backend (AMD MI300A):**
- Baseline: 47ms (no custom pass)
- Custom preprocessing pass `LoopifyInsertSliceChain` at tensor/linalg level
- Generates `flow.dispatch.region { scf.forall(327680) + scf.for(90) }` with WorkgroupMappingAttr
- **Status:** structurally correct, correctness bug in temperature field (root cause identified, fix planned)

## Gap Analysis: Where Do the Remaining 23ms Go? (MI300A)

Based on nsys profiling at the 35ms stage on GH200 (MI300A breakdown not yet profiled):

| Component | Est. Time (ms) | Notes |
|:---|:---:|:---|
| q_t_update kernel | ~8-10 | 2 fused kernels (down from ~80) |
| Precipitation scan (injected) | ~8-10 | Unrolled but fused StableHLO |
| Kernel launch overhead | ~3-5 | ~186 kernels still present for precip |
| Other overhead (scheduling, memory) | ~2-3 | To be profiled |

**To reach <10ms**, the precipitation scan must become 1-2 kernels (as in the XLA SerialScan pass) AND q_t_update fusion must be further improved. An nsys profile on MI300A with the current 23ms result is needed for a precise breakdown.

## Risks and Open Questions

### Transpose overhead (45ms on MI300A)

The 23ms result assumes pre-transposed `(nlev, ncells)` layout. Adding transposes at runtime costs 45ms (total 68ms), making it **slower than doing nothing**. In production, the data must either:
- Be stored in `(nlev, ncells)` layout natively (requires upstream changes)
- Be transposed once at initialization (amortized over many timesteps)
- Be eliminated by a compiler pass (XLA does not do this automatically; investigated, not promising)

### IREE pass correctness bug

- **Root cause identified:** iter1 boundary construction uses all iter0 values instead of only slice-dependent ones
- **Fix:** port XLA's `depends_on_slice` forward propagation — the algorithm is understood and the code change is localized (~50 lines)
- **Timeline:** days, not weeks

### XLA pass requires building from source

- Building JAX + XLA from source takes ~2-4 hours (first build), ~15-30 min incremental
- The pass modifies 5 XLA core files (small, well-documented patches)
- Risk: XLA API changes across versions may break the patches
- Mitigation: pin to a specific JAX/XLA version; patches are documented in [INTEGRATION.md](../xla_passes/INTEGRATION.md)
- Long-term: propose upstreaming to XLA if the approach proves general

### IREE CUDA memory errors

- JIT-compiling the full fused graupel function on IREE CUDA hits memory errors
- Workaround: split into separately JIT-compiled functions (adds overhead)
- IREE's CUDA backend receives limited upstream attention; HIP/ROCm is better supported
- **Status:** low priority, not actively investigated

### JAX version regression (0.6.0 → 0.9.2: 13ms → 23ms)

- Same code, same GPU (MI300A/Beverin), same StableHLO module: **13ms on JAX 0.6.0 vs 23ms on JAX 0.9.2** (~79% regression)
- Root cause unknown — likely XLA compiler changes between versions affecting fusion, scheduling, or memory allocation
- This means performance is fragile and version-dependent; pinning JAX versions for production is essential
- Need to investigate which JAX/XLA change caused the regression and whether it can be worked around
